# Gotchas — the mistakes that break Verse code

Quick-lookup list of where Verse departs from imperative habits. These are the things an LLM gets wrong by writing "what feels right." If you're about to write something that *looks* obvious, check here first.

## Calls & failure

- **`()` vs `[]`.** Call total functions and access tuples with `()`; call `<decides>` functions and index arrays/maps/casts with `[]`. Using `()` on a `<decides>` function (or `[]` on a total one) is a compile error.
- **No booleans for control.** Expressions **succeed or fail**. `if (X > 0):` works because `X > 0` *fails* when false. `if (SomeLogic)` is wrong unless it's failable — use `if (SomeLogic?)`.
- **Comparisons return their left operand**, not a bool: `X := (0 < 10)` makes `X = 0`. They're failable. Relational ops (`< <= > >=`) sit at **precedence level 6 and are right-associative**; `A <= B <= C` is special chain syntax checking both and returning `A`.
- **`<decides>` must be paired with a heap effect (current build).** Write `<decides><computes>` (pure) or `<decides><transacts>` (mutating). A bare `<decides>` is uncallable from failure/rollback contexts. Same for any helper called from a `<decides>`/`<transacts>` body — annotate it `<computes>`/`<transacts>` even if it can't fail.
- **`<no_rollback>` contexts.** Scene Graph lifecycle methods, event `Signal`, and `spawn`/`branch` bodies don't roll back — a method that signals an event **can't be `<decides>`**; return `logic` instead.
- **`if`/`for` interiors are rollback contexts — no_rollback calls are banned there (3512).** You can't CALL a signalling/plain function inside an `if` condition or `for` domain, even just to test its `logic` result: `if (Wallet.Add(N)?)` won't compile. Hoist: `Added := Wallet.Add(N)` then `if (not Added?)`. Same reason a helper iterated by `for (X : Helper())` must carry `<transacts>`/`<reads>`.
- **Interface conformance pins effects.** Implementing an interface means matching its declared effects, and an override can never add no_rollback — e.g. engine `cancelable.Cancel()<transacts>` cannot be implemented by a `Cancel` that signals an event. When your type can't conform, don't implement the interface; type the variables concretely.
- **Two interfaces, same member name → two qualified implementations.** `class(a_iface, b_iface)` where both declare `Enable()` needs `(a_iface:)Enable<override>()` AND `(b_iface:)Enable<override>()` — one shared method does not satisfy both. Delegate to a common private helper.
- **Failable expr needs a failure context.** `Arr[0]`, `Map[K]`, `A/B`, `Type[V]`, `Opt?`, comparisons must sit in `if(...)`, a `for`/`first` filter, a `<decides>` body, `option{}`, `logic{}`, `not`, or `or`. Bare `Arr[0]` in a `void` function won't compile.
- **Don't launder failure early.** Prefer `<decides>` helpers that propagate over `?t`/`logic`/`or Default` returns — converting forces callers to re-test and severs rollback. Consume with `if` at the real decision point; convert only where a value is forced (stored field, `@editable`, before `<suspends>`, the `<no_rollback>` frontier).
- **`<suspends>` + `<decides>` is forbidden.** Hide failure (`option{Failable[]}`) before suspending.
- **Constructors are never `<suspends>`.** Do async setup in `OnBegin`/`OnSimulate`, not construction.
- **Class construction has the `<transacts>` effect (it allocates)** — so `archetype{}` instantiation inside a `<computes>` function is 3512, even for a pure-data class. Fixes: give the factory `<transacts>`, or declare the class itself `class<computes>` (as with `<computes>` error hierarchies) when it qualifies. Struct construction is a value operation and stays fine in `<computes>`.

## Numbers

- **`int / int → rational`**, and `/` is `<decides>` (fails on 0). Get back to `int` with **`Floor(R)`** — *parens* (total on rational). `Floor[F]` with *brackets* is the float overload (`<decides>`). `Int[Floor(R)]` is always wrong (`Floor(R)` is already `int`; `Int[]` only takes `float`).
- **`Floor(A / B)` in HOT paths is an allocation bomb**: the rational is arbitrary-precision and heap-allocated per call. In per-node/per-tick code use **`Quotient[A, B]` and `Mod[A, B]`** (int-native, no rational). Measured in practice: coordinate math (`Floor(Sq / 8)`) running thousands of times per game-tree node dominated the profile.
- **`/=` is float-only** (integer division is failable). Use `set X = Floor(X / N)` for ints.
- **ALL arithmetic is same-type-only — mixed `int`/`float` operands never compile.** `Count / 10.0`, `X + 1` (X float), `N * 0.5` are all errors, not conversions: `+ - * /` only have `(int,int)` and `(float,float)` overloads. Promote the int first: `(1.0 * Count) / 10.0`. There is **no implicit int→float and no `Float()`/`ToFloat()`**. Recognize it in errors: 3509 "No overload of operator'/' matches (:int, :type{10.000000})" — a `type{<number>}` operand means a numeric literal of the *other* type.
- **`0 = 0.0` fails** (cross-type). Promote first. `"5" = 5` fails too.
- **NaN:** `NaN = NaN` is *true* in Verse; there's one NaN; `0.0` has no sign. Arithmetic that would produce NaN (`0.0/0.0`, `Inf - Inf`) is a **runtime error**, but math functions (`Sqrt(-1.0)`, `Ln(-1.0)`) **return** NaN. Float `/0.0` → `Inf` (no failure); int `/0` fails.
- **`Round` is banker's rounding** (ties to even): `Round[0.5]=0`, `Round[2.5]=2`, `Round[-1.5]=-2`.
- **Don't shadow intrinsics** (`Max`, `Min`, `Floor`, `Clamp`, `Abs`, `Mod`, …) — causes "ambiguous identifier". And intrinsics aren't first-class (can't store `Abs` in a var).

## Control flow

- **`for` is an expression returning an array.** No `break`/`continue`/`return` inside it. Filter in the domain (`for (X : Arr, Pred[X])`); for early-exit-on-first-match use **`first`**; for accumulate-until use `loop` + a `var`.
- **A failing `for` filter OR body SKIPS that iteration — it never fails the enclosing function.** Wrong tool whenever one element failing should abort the whole operation: `for (A : Args, V := Eval[A]) {...}` silently drops the failing arg and "succeeds" (this shipped a real bug: an interpreter's `print(unknown_name)` printed nothing instead of erroring). All-must-succeed forms: count-check afterwards (`Results.Length = Inputs.Length`), or accumulate in a plain sequential statement per element outside `for`.
- **`loop` is the only construct with `break`** (which takes no args and has bottom type). Every statement in a `loop:` body must succeed — wrap `<decides>` ops in `if(...)`. `loop` itself returns the top type `true`.
- **The runtime also kills long uninterrupted compute** ("Computation timed out", same unrecoverable ErrRuntime_InfiniteLoop). Any heavy `<transacts>` algorithm (search, solver) must be sliced: a `<suspends>` driver runs bounded slabs with `Sleep(0.0)` between them. Since `<suspends>` can't combine with `<decides>`, slice at natural work boundaries (per root move, per item) and make the slab-internal recursion total.
- **Fuel/budget counters must NOT be `<decides>` checks inside rollback-heavy recursion.** A failing budget check fails the subtree → STM rolls back the counter increments themselves → a caller's for-skip retries the next sibling against a REFRESHED budget → unbounded work. Fuel-as-failure only works when failure aborts the whole computation (minilang). In searches with sibling alternatives, use a TOTAL abort flag + sentinel returns; discard aborted slabs whole.
- **Iterate by recursion when failure must propagate — but the VM caps the CALL STACK at ~120 total frames.** `ErrRuntime_InfiniteLoop: "Infinite script recursion (120 calls) detected"` is an UNRECOVERABLE runtime kill, not a failure — and measurement shows it's ~120 **total Verse frames**, not per-function: a mutually-recursive 4-function cycle (an interpreter's Eval/EvalCall/ExecBlock/ExecStmt) dies after ~30 logical levels. So: recursion-as-iteration only where depth is structurally small (parser rules over one line, log-depth splits, balanced-tree walks); data-proportional depth must use `loop` + option-captured failure (`MaybeR := option{Step[]}`, re-raise after with `not Failed?`); and guard user-driven recursion with a depth counter budgeted as **120 / host-frames-per-logical-level, with margin** (an interpreter at ~5 frames/call gets a cap of ~20).
- **`first`** returns the first body value reaching it, fails if none (`<decides>`); `first{ E : Gen }` pulls one item from a generator without building an array.
- **Flatten multi-clause `for`/`if`; use `for`'s array result.** Prefer the block forms `for:`/`do:` and `if:`/`then:`/`else:` over cramped inline `for (A, B:=…, filter):` / `if (A, B:=…):` — put each clause on its own line. Nested `for`s collapse into one multi-source `for` (flat-map): `for: A:Xs; B:F(A) do: B`. Since `for` returns the array of body values, write `for(…): X` instead of `var Acc; for(…): set Acc += array{X}` — keep the accumulator only for non-1:1 bodies (conditional appends, side mutations). Single-condition `if (X):` one-liners are fine as-is.
- **No iterative `sync`/`race`/`rush`.** `race(E : Gen){...}` doesn't exist. `rush`/`branch` can't even be used directly inside `loop`/`for` bodies (wrap in a function). For a dynamic fan-out, recurse with binary split (see `patterns.md`).
- **Ranges are ascending, `for`/`first`-only, bound with `:=`.** `5..1` is empty; reverse via `for (I := 0..N-1): Use(N-1-I)`. Collection iteration uses `:` (`for (X : Arr)`), range uses `:=` (`for (I := 0..N-1)`).
- **`case` doesn't support `float`, objects, or tuples** — only `int`/`logic`/`char`/`string`/enums/refinements. Open enums always need a `_` wildcard or `<decides>` context.
- **A bare `{}` is not an empty expression** — as a do-nothing statement/arm write `block {}`. `{}` is only valid where a block *argument* is expected: `if (Cond) {}`, `F():void = {}`, archetypes (`type{}`).
- **`defer` does NOT run on failure-rollback** (it's as if it never registered); it can't `<suspends>`/`<decides>`/`return`/`break`; runs LIFO and on cancellation.

## Mutability & bindings

- **`set L = R` requires R to succeed.** Don't put a failable projection on the RHS (`set Arr[I] = Arr[I] - V` is wrong because `Arr[I]` is failable). Use compound assignment: `set Arr[I] -= V`.
- **`set` returns the RHS value** (chainable: `set Y = set X = 5`). Compound assignment evaluates the LHS **once** (unlike the expanded form).
- **No variable shadowing anywhere** — not even nested blocks. Re-binding a name is an error; use `set` for reassignment. This includes **parameters vs module-scope functions**: a param named `MoveText` is ambiguous (3532) with a module function `MoveText(...)`. Rename the param or qualify.
- **Reading a `var` returns an immutable snapshot.** `Y := X` copies; later `set X = ...` doesn't change `Y`. Classes are the exception (reference/shared).
- **Struct-in-array = independent copy; class-in-array = shared reference.** `set A[0].Field = v` changes all class elements but only one struct element.
- **Set array element can't grow it** (`set A[i]=v` fails if `i>=Length`); append with `set A += array{v}`. Map `set M[k]=v` adds-or-updates (the insert itself is infallible but **still needs a failure context** — `if (set M[K]=V):` / `option{set M[K]=V}`); `set M[k]+=v` fails if `k` absent, so update-or-create is `if (set M[K]+=V): else if (set M[K]=V):`. Maps have **no delete** — rebuild via `for`.
- **`+=`/`-=`/`*=`//`=` are NOT overloadable and don't use your `operator'+'`.** They resolve only against builtins (int, float, `[]t + []t`, map insert). "No overload of operator'+='" (3509) on an array append means the element types no longer unify — e.g. pushing `subscription_task` into `[]cancelable` after dropping that conformance. Fix: retype the variable concretely (an operator can't help).
- **Parametric-type toolchain bugs (current build) — the workaround set:**
  - **3502 "Can't access a function from a preceding type" / 3506 unknown identifier across siblings**: a parametric type used generically (implemented, extended, instantiated with `t`) from the **same module level** as its definition fails — and a module hosting parametric definitions gets its member resolution **poisoned for everyone else**: siblings and submodules stop resolving even its plain non-parametric members via `using` (3506). **NEVER put parametrics at the content-root module** — root members are every top-level module (`Stats`, `Currency`, ...), so root-level parametrics break `using { Stats }` project-wide with baffling 3506s in unrelated files. Robust layout: ONE file declaring a dedicated wrapper module (`Collections<public> := module:`) with the explicit hierarchy inside — parametric interfaces + type aliases at wrapper level, `Containers := module:` with implementing classes under it, `Ops := module:` nested for extensions. Also keep a parametric class to ONE parametric superinterface — combine several into one interface first (`list_view(t) := interface(graph(t), stack(t), queue(t))`). Known upstream: "Parametric types definition ordering", <https://github.com/verselang/book/discussions/27> — maintainer-confirmed compiler **pass-ordering** issue ("no fundamental reason", fix is non-trivial). Extra hazards from that thread + practice: UEFN can HIDE the error and **crash the preview session on start**, and **build/LSP state goes stale in BOTH directions** — the LSP misses what the in-editor build catches, and the build can re-report errors whose fix is already on disk (byte-identical error text to a previous wave is the tell). Before debugging a "surviving" error, rebuild fresh; if it persists, restart the editor. Only then treat it as real.
  - **3502 "Failure in the right operand of 'set' is not yet implemented"**: `set X = X.Slice[...]` (failable RHS) doesn't compile even in a failure context. Hoist: `Shrunk := X.Slice[...]` then `set X = Shrunk`.
  - **3509 "expects t, but argument is an incompatible value of type t"** on constructing a parametric class: a parametric class used as a **field type of another parametric class** (`weighted_edge(t)` stored in `weighted_node(t).Out`) mis-unifies its type param at construction (`__dupe___unnamed_parameter_1:t`). Fix: flatten to **parallel arrays** on the owner (`OutTargets:[]node(t)` + `OutCosts:[]float`) or restructure to avoid the nested parametric field.
  - **3552 "implicit type parameter used in the type of mutable data may not be used in the type of a function"**: ANY `t`-generic function — extension OR free function — whose signature mentions a parametric class that has `var` fields typed by `t` (`FindPath(G:weighted_digraph(t)...)` where `Members:[]weighted_node(t)` is `var`; list ops over `linked_list(t)` with var `Head`). Fixes: make fields non-`var` where possible; otherwise the operation must be a **member function on the owning class** (members are exempt). Consequence: algorithms over mutable parametric containers live as members, not extensions. When the moved-in member needed an extension-only constraint (`t:subtype(comparable)` for value equality), **inject the capability as a function argument** instead — `IndexOf(V:t, Equal:equality(t))` with `equality(t) := type{_(:t,:t)<transacts><decides>:void}` — same pattern as comparator injection. Extensions over **interfaces** (`graph(t)`, `binary_node(t)`) and over plain `[]t` stay safe: no var fields in sight.
  - **Definition order matters even INSIDE one module**: parametric support types (`path_entry(t)`, `path_frontier(t)`) must appear textually **above** the parametric class whose members use them. Lenient evaluation does not rescue parametric ordering — write parametric files strictly top-down.
  - **3503 "Type variable t is unused"** on zero-arg extensions: `(L:list(t) where t:type).IsCyclic()` has `t` in no argument position. Fix: make it a member function (or add a `t`-typed parameter).
  - **3510 covariance not applied to returns**: `[]derived(t)` is not accepted where `[]base(t)` is declared, despite array covariance. Fix: rebuild with explicit upcasts — `GN:base(t) = D` then `set R += array{GN}`.
  - **3532 ambiguous monomorphic extension overloads**: same-named extensions on `[]int`/`[]float`/`[]string` are ambiguous at an `array{}` call site — give them distinct names (`SortedInts`/`SortedFloats`/`SortedStrings`).
  - **No `<`/`<=` on `string`/`[]char` or `char`** (3509: only int/float overloads exist despite docs claiming string ordering). Char classes via membership in a set string (`first (X : "0123456789", X = C) { X }`); string ordering via manual lexicographic compare ranking chars against a reference alphabet.
  - **Extension names must be unique across receiver types** — `(N:node(t)).Depth()` and `(T:tree(t)).Depth()` collide; name them `NodeDepth`/`TreeDepth`.
- **Bindings in `and`/`or`/`not` don't escape** to the `then`-branch; but `if (X := Arr[0]):` does bind `X` in `then`.
- **`var` needs an explicit type** (`var X := 0` is an error); module-scope mutable state must be a `weak_map`.
- **Required `@editable` references on COMPONENTS: leave the field uninitialized** (`Board:entity`, `Key:string`) — components aren't `<concrete>`, and UEFN forces designers to set defaultless editables, so `?entity = false` there just trades a compile-time guarantee for runtime unwrap noise. The optional-with-default shape is for **`<concrete>` classes instantiated inline in the details panel** (strategy objects, presenter configs), where `<concrete>` demands every field have a default.
- **Field defaults reject divergent calls (3582).** A data-member initializer must provably terminate; any call whose body loops (`for`/`loop`) — i.e. anything not `<converges>`/`<computes>`-trivial — is "divergent" and banned: `var Current:chess_position = StartingPosition()` fails. Fixes: leave the field default-less behind a factory (`MakeGame()<transacts> = game{Current := Start()}`, class `<internal>` for the constructor-accessibility rule), or lazy-init via `?type = false` + an `Ensure()` accessor when the owner is a component (whose construction you don't control).

## Types & containers

- **Tuple access `T(0)` is parens, compile-time, never fails. No destructuring binding** `(A,B) := T` — bind then index `T(0)`, `T(1)`. *(But function **parameters** can destructure: `F(A:int, (B:int, C:int))`.)*
- **`?T` not `T?`** for the type. Empty option is `false` (not null). `option{e}` wraps `<decides>`→`?t`; `o?` unwraps; `o? or Default`; `o?.Field` short-circuits.
- **`;` vs `,`** differ by context: in a tuple/`option{}`/`logic{}`, `,` builds a tuple and `;` sequences (last value). `array{}` accepts either but **can't mix** them. In `for`, both separate filters.
- **`string` length is bytes** (`"José".Length = 5`); `string = []char` of UTF-8 code units. `char` (0o.. literal, 1 byte) vs `char32` (0u.. literal) — no implicit conversion.
- **Interpolation `{X}` only takes ToString-able types: `int`/`float`/`char`/`string`.** Arrays, maps, options, logic, tuples, objects all fail with 3509 "No overload of ToString". Format them yourself: `var Line:string = ""; for (V : Arr): set Line = "{Line}{V} "`.
- **Classes aren't comparable unless `<unique>`** (then by identity). Structs compare field-by-field if all fields comparable.
- **`weak_map` is a *supertype* of `map`** — assigning a `map` to a `weak_map` loses `.Length`/iteration. Module-scoped `weak_map` only supports per-key read/write.
- **Generators (`Find*` results) aren't arrays.** No indexing, single iteration. Take one with `first{...}`; materialize with `for (X : Gen){X}`.

## Scene Graph / engine specifics

- **Two SpatialMath namespaces.** `/Verse.org/SpatialMath` `vector3` is **Forward/Left/Up**; `/UnrealEngine.com/Temporary/SpatialMath` `vector3` is **X/Y/Z** (used by `fort_character.GetTransform/GetViewRotation`). They're different types — convert with `FromTransform`/`FromRotation`/`FromVector3`. `vector2` only exists in the Temporary namespace.
- **`GetComponent[...]` is `<decides>`** — use it in a failure context, and the caller needs a heap effect.
- **One component per subclass-group per entity.** Need two lights → two entities.
- **Subclass components with `<final_super>`**; prefabs subclass `entity`. Keep logic in components, not the entity subclass.
- **Always call `(super:)OnBeginSimulation()` etc.** when overriding lifecycle methods, and cancel subscriptions in the matching teardown method.
- **No polling tick loops.** `loop: Check(); Sleep(0.1)` is the imperative habit leaking back in — be event-reactive: `Await` events, express gameplay over time with `race`/`sync`/sequencing. `Sleep`-cadence only when the effect itself is periodic; frame-coupled work uses `tick_events`.
- **Can't attach entities/components to an `agent`** (currently) — hanging components/child entities off a player needs the state-entity proxy workaround. Plain per-player *data* doesn't: a module `weak_map(session, [player]t)`, a manager-component map field, or a persistent `weak_map(player, persistable)` are all normal homes (`patterns.md` §4).
- **`agent.GetFortCharacter[]` is `<transacts><decides>`** and the character **changes on respawn** — re-fetch and re-subscribe; don't cache a `fort_character` across deaths.
- **Health clamps to `[1.0, MaxHealth]`** via `healthful.SetHealth`; you can't `SetHealth(0)` — eliminate through `damageable.Damage`.

## Modules & evolution

- **Module bodies hold definitions only** (no executable statements) and require explicit type annotations (no `:=` at module scope).
- **`Content/Collections` is not usable as a Verse module folder** — it's UE's reserved asset-collections directory (always present, empty-looking). Name code folders something else (`DataStructures/`). Same caution for other editor-reserved roots (`Developers/`, `__ExternalActors__/`, ...).
- **`using` is a statement** at file/module top level; each file needs its own imports (same-folder files see each other without imports). Import parents before nested children.
- **Undeclared folder modules default to package-internal**, which surfaces as 3593 ("definition accessible universally, but depends on X, only accessible from subpaths of /<package>") the first time a `<public>` definition in a *declared-public* module exposes one of its types (e.g. a public `@editable` field typed by it). Fix: declare every folder module whose types appear in public APIs as `<public>` in modules.verse. The general 3593 rule: a definition must be no more accessible than every type in its signature.
- **Publishing is a permanent contract.** Once public, you can't remove a definition, change its kind, narrow access, or widen effects/param-narrow/return-widen on final members; non-final instance members must keep exact signatures. `<castable>`/`<final_super>` are irreversible. Persistable schemas can only add defaulted fields / open-enum values. Over-specify effects (`<reads>` you might later drop to `<computes>`).
